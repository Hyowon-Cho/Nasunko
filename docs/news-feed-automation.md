# 나선코 뉴스 자동화 성능 개선 및 데이터 연결 디버깅 기록

## 1. 배경

나선코는 나스닥, 빅테크, 반도체, 환율, 금리, 유가 관련 뉴스를 한 화면에서 추적할 수 있도록 만든 금융 정보 서비스입니다. 뉴스 기능은 외부 RSS 뉴스 소스를 주기적으로 수집하고, 수집한 기사를 RDS PostgreSQL에 저장한 뒤, 사용자가 뉴스 페이지에 접속하면 DB에 저장된 최신 기사를 조회해 보여주는 구조입니다.

초기에는 뉴스와 경제지표를 하나의 피드 화면에 함께 배치했습니다. 하지만 실제 운영 과정에서 두 가지 문제가 나타났습니다.

- 외부 RSS 뉴스 소스 응답이 느릴 때 뉴스 페이지 또는 뉴스 수집 API가 지연될 수 있었습니다.
- FRED 경제지표 API가 실패하거나 캐시된 실패 결과가 남으면 화면에 `데이터 없음`이 반복적으로 노출되었습니다.

이 문제를 해결하기 위해 뉴스 수집 자동화, DB 저장 방식, 외부 데이터 의존성, 화면 구성 범위를 함께 점검했습니다. 최종적으로 뉴스 페이지는 뉴스 데이터에 집중하도록 정리하고, 경제지표 영역은 제거했습니다.

## 2. 문제 정의

### 문제 1. 외부 뉴스 소스 응답 지연

뉴스 수집이 느려지는 가장 큰 원인은 외부 RSS 뉴스 소스의 응답 시간이 일정하지 않다는 점이었습니다.

측정 결과, 일부 RSS 소스는 빠르게 응답했지만 특정 소스는 한 번 호출하는 데 약 4.7초까지 걸렸습니다.

예시 측정 결과:

- 한국경제 RSS: 약 0.5초
- 연합뉴스 RSS: 약 1.8초
- 동아일보 RSS: 약 4.7초

뉴스 페이지가 DB에서 최신 뉴스를 찾지 못하면 fallback으로 RSS를 직접 호출하는 구조였기 때문에, 외부 RSS가 느릴 경우 사용자가 체감하는 페이지 응답 시간도 함께 느려질 수 있었습니다.

### 문제 2. 뉴스 저장 방식의 비효율

기존 뉴스 수집 로직은 수집한 기사를 하나씩 반복하면서 개별 INSERT 쿼리를 실행했습니다.

즉, 60개 이상의 기사를 수집하면 데이터베이스 INSERT 요청도 기사 수만큼 발생할 수 있는 구조였습니다.

이 방식은 구현은 단순하지만 다음 문제가 있습니다.

- 데이터베이스 왕복 요청이 많아집니다.
- 수집 기사 수가 많아질수록 처리 시간이 선형적으로 증가합니다.
- Vercel 서버리스 환경에서는 외부 요청과 DB 요청이 누적되어 timeout 위험이 커집니다.

### 문제 3. 자동 수집 상태 확인이 어려움

DB를 확인했을 때 뉴스 데이터가 과거 날짜에서 멈춰 있었습니다. 로직 자체가 완전히 깨진 것은 아니었지만, 자동 수집이 실제 배포 환경에서 꾸준히 실행되지 않았을 가능성이 있었습니다.

확인 결과 로컬에서는 뉴스 cron API가 정상 작동했고, 수동 실행 시 최신 뉴스가 DB에 저장되었습니다. 따라서 코드 문제와 별개로 GitHub Actions 또는 Vercel 환경변수의 `CRON_SECRET` 설정도 점검해야 하는 상황이었습니다.

### 문제 4. 경제지표 API 실패가 뉴스 화면 품질까지 낮춤

뉴스 페이지에 경제지표 카드를 함께 배치하면서, 뉴스 기능과 FRED 경제지표 API가 같은 화면에서 결합되었습니다.

FRED API 자체는 정상 응답할 때 숫자를 제공했지만, 배포 환경에서는 다음 문제가 생길 수 있었습니다.

- API 키 누락 또는 환경변수 미반영
- 외부 API 일시 실패
- 이전 실패 결과가 캐시에 남아 `데이터 없음` 반복 노출
- 뉴스 화면인데 경제지표 오류가 먼저 눈에 들어오는 UX 문제

결과적으로 뉴스 자동화는 정상이어도 사용자는 화면 상단의 `데이터 없음` 때문에 전체 데이터 연결이 실패한 것처럼 느낄 수 있었습니다.

## 3. 원인 분석 과정

문제를 단순히 화면 오류로 보지 않고 데이터 흐름을 단계별로 나눠 확인했습니다.

1. RDS PostgreSQL 연결 확인
   - DB ping 정상
   - `news_articles` 테이블 조회 정상
   - `news_sync_runs` 실행 기록 조회 정상

2. 최신 뉴스 데이터 확인
   - 기존에는 최신 뉴스가 오래된 날짜에 머물러 있었습니다.
   - 수동 cron 실행 후 최신 뉴스가 정상 저장되는 것을 확인했습니다.

3. 외부 RSS 응답 속도 측정
   - RSS 소스별 응답 시간을 직접 측정했습니다.
   - 특정 RSS 소스가 전체 수집 시간을 늘리는 병목임을 확인했습니다.

4. 뉴스 수집 로직 확인
   - 수집된 기사를 DB에 하나씩 INSERT하고 있었습니다.
   - 반복 DB 요청이 성능 저하 요인이 될 수 있음을 확인했습니다.

5. 뉴스 페이지 응답 확인
   - DB에 최근 뉴스가 있으면 페이지는 빠르게 응답했습니다.
   - DB에 최근 뉴스가 없을 때만 외부 RSS fallback이 발생하므로, 자동 수집 안정성이 중요했습니다.

6. 경제지표 영역 분리 판단
   - `getIndicators()` 함수 자체는 FRED에서 숫자를 만들 수 있음을 확인했습니다.
   - 하지만 뉴스 페이지에서 경제지표까지 함께 보여주면 외부 API 하나의 실패가 뉴스 페이지 전체 신뢰도를 떨어뜨렸습니다.
   - 따라서 뉴스 페이지는 뉴스만 보여주도록 범위를 축소했습니다.

## 4. 해결 방법

### 해결 1. RSS 요청 timeout 단축

외부 RSS 요청 timeout을 4000ms에서 2500ms로 줄였습니다.

수정 파일:

- `lib/news.ts`

기존에는 느린 RSS 소스를 최대 4초까지 기다렸습니다. 이를 2.5초로 줄여 특정 뉴스 소스 하나가 전체 수집 작업을 오래 붙잡지 않도록 했습니다.

이 변경으로 일부 소스가 느리거나 불안정해도 전체 서비스 응답이 과도하게 지연되는 위험을 줄였습니다.

### 해결 2. 기사 저장 방식을 bulk INSERT로 개선

기존에는 기사마다 개별 INSERT 쿼리를 실행했습니다. 이를 한 번의 bulk INSERT 쿼리로 변경했습니다.

수정 파일:

- `lib/news-sync.ts`

변경 전 구조:

```text
기사 1개 -> INSERT
기사 1개 -> INSERT
기사 1개 -> INSERT
...
```

변경 후 구조:

```text
수집 기사 목록 생성
-> VALUES 목록을 한 번에 구성
-> INSERT ... VALUES (...), (...), (...) ON CONFLICT DO NOTHING
```

이 방식은 DB 왕복 횟수를 크게 줄이고, 중복 뉴스 제거 로직도 그대로 유지할 수 있습니다.

### 해결 3. 최신 뉴스 조회용 인덱스 추가

뉴스 페이지는 최신 뉴스 순으로 데이터를 조회합니다. 이에 맞춰 `published_at DESC` 인덱스를 추가했습니다.

수정 파일:

- `lib/news-sync.ts`

추가한 인덱스:

```sql
CREATE INDEX IF NOT EXISTS news_articles_published_at_idx
ON news_articles (published_at DESC);
```

이를 통해 최신 뉴스 목록 조회 쿼리가 더 안정적으로 동작하도록 개선했습니다.

### 해결 4. 자동 수집 실행 기록 검증

수동으로 `/api/cron/news`를 실행해 수집 파이프라인이 실제로 살아 있는지 확인했습니다.

확인 결과:

- RSS 소스 3개 호출 성공
- 수집 실패 소스 0개
- 중복 기사 정상 처리
- 최신 뉴스 DB 저장 정상

따라서 애플리케이션 로직은 정상이며, 자동 실행이 멈춘 경우에는 GitHub Actions와 Vercel 환경변수 설정을 확인해야 한다는 결론을 얻었습니다.

### 해결 5. 뉴스 페이지에서 경제지표 영역 제거

뉴스 페이지에서 경제지표 카드를 제거하고, 페이지 제목과 설명을 뉴스 중심으로 변경했습니다.

수정 파일:

- `app/feed/page.tsx`
- `components/Header.tsx`
- `components/Footer.tsx`

변경 전:

```text
피드
경제지표 + 뉴스
FRED API 실패 시 데이터 없음 노출
```

변경 후:

```text
뉴스
뉴스 카테고리 + 뉴스 목록
FRED API 의존성 제거
```

이 변경으로 뉴스 페이지는 뉴스 DB 조회에만 집중하게 되었고, 경제지표 API 실패가 뉴스 화면 품질에 영향을 주지 않도록 분리했습니다.

## 5. 개선 결과

개선 전에는 뉴스 수집 실행 시간이 외부 RSS 지연과 개별 INSERT의 영향을 크게 받았고, 경제지표 API 실패가 뉴스 화면의 신뢰도까지 낮추는 문제가 있었습니다.

개선 후 확인 결과:

- 뉴스 수집 API 정상 실행
- 68건 수집
- 68건 중복 처리
- 실패 소스 0개
- 내부 처리 시간 약 436ms
- DB 기반 뉴스 페이지 응답 시간 약 0.19초
- 뉴스 페이지에서 FRED API 의존성 제거
- Next.js production build 통과

특히 DB에 최신 뉴스가 정상 저장된 상태에서는 뉴스 페이지가 외부 RSS를 직접 기다리지 않고 PostgreSQL을 조회하기 때문에 빠르게 응답했습니다.

## 6. 정량 개선 지표

이번 개선은 완전한 A/B 테스트 환경에서 측정한 값은 아니지만, 실제 실행 로그와 코드 구조를 기준으로 다음과 같이 정량화할 수 있습니다.

### 6.1 DB INSERT 요청 수 감소

기존 구조에서는 수집 기사마다 개별 INSERT 쿼리를 실행했습니다. 68건의 기사를 처리하면 최대 68번의 INSERT 요청이 발생할 수 있었습니다.

개선 후에는 수집된 기사 목록을 한 번의 bulk INSERT 쿼리로 처리하도록 변경했습니다.

```text
개선 전: 기사 68건 기준 최대 68회 INSERT 요청
개선 후: 기사 68건 기준 1회 bulk INSERT 요청
감소율: 약 98.5% 감소
```

계산식:

```text
(68 - 1) / 68 * 100 = 98.5%
```

이 개선은 데이터베이스 왕복 비용을 줄이는 데 직접적인 효과가 있습니다.

### 6.2 외부 RSS 대기 상한 감소

기존에는 RSS 소스 하나가 최대 4000ms까지 응답을 지연시킬 수 있었습니다. 이를 2500ms로 줄여 느린 외부 뉴스 소스가 전체 수집 작업을 오래 붙잡지 않도록 했습니다.

```text
개선 전: RSS 소스당 최대 4000ms 대기
개선 후: RSS 소스당 최대 2500ms 대기
감소율: 37.5% 감소
```

계산식:

```text
(4000 - 2500) / 4000 * 100 = 37.5%
```

이 변경은 외부 RSS 소스의 응답 지연이 서비스 전체 지연으로 이어지는 위험을 줄이기 위한 조치입니다.

### 6.3 뉴스 수집 처리 시간 단축

실제 실행 로그 기준으로, 개선 전후 뉴스 수집 처리 시간은 다음과 같이 확인했습니다.

```text
개선 전 실행 로그: 약 1661ms
개선 후 실행 로그: 약 436ms
단축률: 약 73.8%
```

계산식:

```text
(1661 - 436) / 1661 * 100 = 73.8%
```

단, 이 값은 외부 RSS 응답 상황과 중복 기사 비율이 완전히 동일하게 통제된 A/B 테스트 결과는 아닙니다. 따라서 자소서나 면접에서는 "실제 실행 로그 기준"이라고 표현하는 것이 정확합니다.

### 6.4 뉴스 페이지 응답 시간 확인

개선 후 DB에 최신 뉴스가 저장된 상태에서 `/feed` 페이지 응답 시간을 확인했습니다.

```text
/feed 응답 시간: 약 0.19초
```

이는 뉴스 페이지가 외부 RSS를 직접 기다리지 않고, PostgreSQL에 저장된 최신 뉴스 데이터를 조회했을 때의 응답 시간입니다.

### 6.5 데이터 최신성 복구

문제 확인 당시 뉴스 데이터는 과거 날짜에서 멈춰 있었습니다. 수동 cron 실행과 수집 파이프라인 검증 후 최신 뉴스 저장을 확인했습니다.

```text
기존 상태: 최신 뉴스가 2026-06-21 기준으로 멈춰 있음
개선 후: 2026-07-16 기준 최신 뉴스 저장 확인
결과: 약 25일 수준의 최신 뉴스 공백 복구
```

### 6.6 외부 API 의존성 축소

뉴스 페이지에서 FRED 경제지표 호출을 제거하면서, 뉴스 화면이 의존하는 외부 데이터 경로를 줄였습니다.

```text
변경 전: 뉴스 DB 조회 + FRED 경제지표 API 호출
변경 후: 뉴스 DB 조회 중심
효과: 경제지표 API 실패가 뉴스 화면에 노출되는 문제 제거
```

이 변경은 단순히 기능을 줄인 것이 아니라, 핵심 사용자 경험을 기준으로 외부 의존성을 분리한 운영 안정화 작업입니다.

### 6.7 자소서용 정량 표현

자소서에는 다음 문장을 사용할 수 있습니다.

> 뉴스 자동화 과정에서 외부 RSS 응답 지연과 기사별 개별 INSERT로 인한 병목을 발견했습니다. RSS 요청 timeout을 4초에서 2.5초로 줄여 외부 소스 대기 상한을 37.5% 낮췄고, 기사별 INSERT 방식을 bulk INSERT로 개선해 68건 수집 기준 DB INSERT 요청 수를 최대 68회에서 1회로 줄였습니다. 실행 로그 기준 뉴스 수집 처리 시간은 약 1.66초에서 0.44초로 약 74% 단축되었고, DB 기반 뉴스 페이지 응답 시간은 약 0.19초로 확인했습니다. 또한 경제지표 API 실패가 뉴스 화면에 `데이터 없음`으로 노출되는 문제를 발견해 뉴스 화면의 책임 범위를 뉴스 데이터로 분리했습니다.

## 7. 자소서용 정리 문장

나선코 프로젝트에서 뉴스 자동화 기능을 구현한 뒤, 실제 운영 과정에서 뉴스 응답 지연과 데이터 갱신 중단처럼 보이는 문제를 발견했습니다. 문제를 해결하기 위해 RDS PostgreSQL, 외부 RSS 응답 시간, cron 실행 기록, 뉴스 조회 로직을 단계별로 분석했습니다. 그 결과 일부 RSS 소스의 응답 지연과 기사별 개별 INSERT 방식이 성능 저하의 원인임을 확인했습니다.

이를 해결하기 위해 RSS timeout을 줄여 느린 외부 소스가 전체 응답을 지연시키지 않도록 했고, 뉴스 저장 방식을 개별 INSERT에서 bulk INSERT로 변경해 데이터베이스 왕복 비용을 줄였습니다. 또한 최신 뉴스 조회 패턴에 맞춰 published_at 인덱스를 추가했습니다. 개선 후 뉴스 수집 API는 68건의 기사를 약 436ms의 내부 처리 시간으로 중복 처리했고, DB 기반 뉴스 페이지는 약 0.19초 수준으로 응답했습니다.

추가로 경제지표 API 실패가 뉴스 화면에 `데이터 없음`으로 노출되는 문제가 있었기 때문에, 뉴스 페이지의 역할을 뉴스 조회로 명확히 분리했습니다. 이 과정에서 단순히 기능을 많이 붙이는 것보다, 사용자에게 안정적으로 전달해야 하는 핵심 데이터가 무엇인지 정의하고 외부 API 의존성을 줄이는 방향으로 제품 구조를 조정했습니다.

이 경험을 통해 단순히 기능을 구현하는 것에서 끝나지 않고, 실제 데이터 흐름을 관찰하고 병목을 측정한 뒤, 데이터 수집 자동화와 데이터베이스 저장 방식을 개선하는 과정을 경험했습니다. 특히 외부 데이터 소스의 불안정성, 서버리스 환경의 timeout 위험, DB 쿼리 효율성, 화면 책임 범위 분리를 함께 고려해 서비스를 안정화했다는 점에서 데이터 분석 및 BI 업무와 연결되는 경험이었습니다.

## 8. 면접에서 말할 수 있는 핵심 포인트

- 외부 RSS 데이터를 자동 수집해 PostgreSQL에 저장하는 데이터 파이프라인을 구현했습니다.
- 수집이 느려지는 문제를 감으로 판단하지 않고, RSS 응답 시간과 DB 쿼리 상태를 직접 측정했습니다.
- 기사별 개별 INSERT를 bulk INSERT로 바꿔 데이터 저장 효율을 개선했습니다.
- 최신 뉴스 조회 패턴에 맞춰 DB 인덱스를 추가했습니다.
- 자동화가 실제로 작동하는지 확인하기 위해 수집 실행 로그를 DB에 저장하고 검증했습니다.
- 경제지표 API 실패가 뉴스 화면 품질을 낮추는 문제를 확인하고, 뉴스 페이지의 책임 범위를 뉴스 데이터로 분리했습니다.
- 단순 웹 기능이 아니라 데이터 수집, 정제, 중복 제거, 저장, 조회까지 이어지는 end-to-end 데이터 자동화 경험을 만들었습니다.
